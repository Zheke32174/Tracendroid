# `signing/`

Local signing material for the rootfs release pipeline. Nothing in this directory is
committed except this README and the (forthcoming, .gitignored) developer keypair you
generate per the runbook.

## Why this directory exists

The on-device rootfs verifier (`ShellRootfsSignatureVerifier`, see `THREAT_MODEL.md
§ 4.5` + `SHELL_REBUILD.md`) reads the public key from
`app/src/main/assets/rootfs/operit-rootfs-pubkey.pem`. The matching **private** key
must never enter the repo:

- It signs the rootfs artifact in CI (the `ROTATION_RUNBOOK.md` flow).
- Local developers generate their own keypair and replace the public asset for local
  testing only.
- Production rotation goes through the runbook — never through an in-repo key.

The public key currently bundled in the APK is a **development-only key** generated
2026-05-12 during the threat-model branch. The matching private key is NOT in the repo
and was not retained anywhere beyond the keygen session that produced the public asset.
Effect: **shipped builds today cannot pass the rootfs signature check** because no
production signing flow has been set up. That is the correct posture — the verifier
refuses unsigned material rather than silently accepting.

## Local development flow

If you want to test the rootfs install path end-to-end locally:

```bash
# 1. Generate a personal dev keypair.
mkdir -p signing
openssl genpkey -algorithm Ed25519 -out signing/dev-private.pem
openssl pkey -in signing/dev-private.pem -pubout \
    -out app/src/main/assets/rootfs/operit-rootfs-pubkey.pem

# 2. Build a rootfs artifact (or use a stub one) and sign it.
ARTIFACT=path/to/rootfs.tar.zst \
SIGNING_KEY=signing/dev-private.pem \
PUBLIC_KEY=app/src/main/assets/rootfs/operit-rootfs-pubkey.pem \
    bash debian/sign.sh

# 3. Update app/src/main/java/com/ai/assistance/operit/shell/ShellRootfsRelease.kt::
#    EXPECTED_SHA256 to the SHA-256 of the signed artifact. The pin must match or
#    the device-side downloader refuses the install.

# 4. Drop your signed .tar.zst + .sig at the artifactUrl() the release file points at,
#    or short-circuit by side-loading directly into context.filesDir/rootfs-staging/
#    for local testing.
```

`signing/dev-private.pem` is intentionally listed in `.gitignore` (see commit history
for when that line landed). Never `git add -f` a private key.

## Production flow

See `docs/ROTATION_RUNBOOK.md`. The production key:
- Is generated by the project maintainer, once.
- Lives in the `ROTATION_RUNBOOK.md`-documented secret store (GitHub Actions Secrets
  `ROOTFS_SIGNING_KEY`).
- Is rotated through a documented procedure that ships a new app version with the new
  public key embedded — there is no runtime "trust this new key now" path.

If you find a private key checked into this repo, that is a bug. Open an issue and
rotate immediately.
