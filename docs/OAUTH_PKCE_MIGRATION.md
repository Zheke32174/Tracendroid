# OAUTH_PKCE_MIGRATION.md

> Scope and execution plan for migrating GitHub OAuth from confidential-client (with `client_secret`) to PKCE (Proof Key for Code Exchange). Resolves `THREAT_MODEL.md § 4.8` once implementation lands. 中文镜像见 [`OAUTH_PKCE_MIGRATION.zh.md`](./OAUTH_PKCE_MIGRATION.zh.md)。

## Why

`app/build.gradle.kts` lines 74–75 embed `GITHUB_CLIENT_SECRET` as a `BuildConfig` field sourced from `local.properties`. Any populated value flows into the APK and is recoverable by decompilation. Per `SECURITY.md` red line **"the APK does not embed secrets"**, this must be removed.

The current OAuth flow is **pure confidential-client**: `client_id` + `client_secret` + `code` → `access_token`. No PKCE machinery exists — zero occurrences of `code_verifier`, `code_challenge`, or `pkce` in the codebase as of branch `claude/operit-fork-optimization-WWapH`. The migration adds the PKCE round-trip and drops the secret in one atomic PR.

GitHub OAuth Apps support PKCE (`response_type=code` with `code_challenge` + `code_challenge_method=S256`) regardless of whether a secret is registered. Users who registered their OAuth App with a secret per the current `docs/BUILDING.md` do not need to re-register — they just stop using the secret.

## Target state

### Authorization URL

Built by `GitHubAuthPreferences.getAuthorizationUrl()`:

```
https://github.com/login/oauth/authorize
  ?client_id=<id>
  &redirect_uri=operit://github-oauth-callback
  &scope=notifications,public_repo,user:email,read:user
  &state=<random32>
  &code_challenge=<base64url-no-pad(sha256(verifier))>
  &code_challenge_method=S256
```

### Verifier generation and persistence

- `code_verifier` is a 64-character URL-safe random string (43–128 per RFC 7636). Generated by a new `PkceCodeGenerator` object alongside `GitHubAuthPreferences` (likely `data/preferences/PkceCodeGenerator.kt`).
- The verifier persists in the existing `githubAuthDataStore` (a new key, e.g. `PENDING_OAUTH_CODE_VERIFIER`, paralleling the existing `PENDING_OAUTH_STATE` key). It's consumed and cleared on first read by the callback handler, exactly like the state token.
- **Encryption deferred**: `PENDING_OAUTH_STATE` already lives in plain DataStore, and the project's broader credential encryption work is tracked in `THREAT_MODEL.md § 4.9`. The verifier follows the same pattern in this PR and migrates to `EncryptedSharedPreferences` alongside the rest of § 4.9.

### Access-token POST

`GitHubApiService.getAccessToken(code, verifier)`:

```kotlin
val formBody = FormBody.Builder()
    .add("client_id", GitHubAuthPreferences.GITHUB_CLIENT_ID)
    .add("code", code)
    .add("code_verifier", verifier)
    .build()
```

The `client_secret` line is removed. The function signature gains the verifier parameter; the coordinator passes the consumed verifier through.

### Removals

- `app/build.gradle.kts:75` — `buildConfigField("String", "GITHUB_CLIENT_SECRET", ...)` line deleted.
- `GitHubAuthPreferences.kt` companion — `val GITHUB_CLIENT_SECRET = BuildConfig.GITHUB_CLIENT_SECRET` line deleted.
- `local.properties.example` — no change required (the example file doesn't currently mention the secret; users following `docs/BUILDING.md` may have added it locally).
- `docs/BUILDING.md` — OAuth setup step rewritten to omit the secret. Add a note: "The Client Secret you may see in the GitHub OAuth App settings is no longer used by the app — PKCE replaces it. Leave the secret in GitHub's UI; this app simply ignores it."

## Step-by-step implementation

1. **Add `PkceCodeGenerator`** at `app/src/main/java/com/ai/assistance/operit/data/preferences/PkceCodeGenerator.kt`. A small `object` with:
   - `generateCodeVerifier(): String` — 64 URL-safe random characters using `SecureRandom`.
   - `computeCodeChallenge(verifier: String): String` — base64url-no-pad SHA-256 via `java.security.MessageDigest` + `android.util.Base64.URL_SAFE or NO_PADDING or NO_WRAP`.
2. **Update `GitHubAuthPreferences.kt`**:
   - Remove `val GITHUB_CLIENT_SECRET`.
   - Add `private val PENDING_OAUTH_CODE_VERIFIER = stringPreferencesKey("pending_oauth_code_verifier")`.
   - Add `setPendingCodeVerifier(verifier: String)` and `consumePendingCodeVerifier(): String?` paralleling the state methods.
   - Change `getAuthorizationUrl(state)` to additionally accept (or generate-and-persist) a verifier, compute the challenge, and add `code_challenge` + `code_challenge_method` to the URL. Signature: `suspend fun getAuthorizationUrl(state: String = createOAuthState()): String` — the suspend modifier is new and allows the verifier to be persisted before returning the URL.
3. **Update `GitHubApiService.kt::getAccessToken`**:
   - Add `verifier: String` parameter.
   - Replace `.add("client_secret", …)` with `.add("code_verifier", verifier)`.
4. **Update `GitHubOAuthCoordinator.kt`** to:
   - Call `consumePendingCodeVerifier()` when handling the callback.
   - Pass the verifier to `getAccessToken(code, verifier)`.
   - Fail the flow visibly (not silently) if the verifier is missing — per `SECURITY.md` "no fallback patterns in security paths".
5. **Remove `GITHUB_CLIENT_SECRET` from `app/build.gradle.kts`**. Delete line 75.
6. **Update `docs/BUILDING.md`** Step 5 prose. Drop secret-related instructions. Note PKCE is in use.
7. **Update `THREAT_MODEL.md § 4.8`** + ZH mirror: status `open → closed`, location refreshed.
8. **Update `AUDIT_PLAN.md § Release gates`** secret-scan row: confirm the gate now passes against the post-migration code.

## Testing plan

1. Local build: `./gradlew :app:compileDebugKotlin` succeeds.
2. Smoke run: install the debug APK, tap the GitHub login button.
3. Browser opens the GitHub authorize URL. Verify it contains `code_challenge=` and `code_challenge_method=S256`.
4. Approve in the browser. Redirect to `operit://github-oauth-callback?code=...&state=...`.
5. App receives the callback. `GitHubOAuthCoordinator` exchanges with PKCE. `access_token` returned.
6. Verify the access token works: pull `GET /user` returns the logged-in user.
7. Decompile the release APK (`apktool d`); grep for `client_secret`; confirm zero hits.

## Out of scope

- **Credential encryption at rest.** The verifier persists in the existing plain DataStore alongside `PENDING_OAUTH_STATE`. Migration to `EncryptedSharedPreferences` happens in the larger `§ 4.9` credential storage PR, not here.
- **Other OAuth providers** (Google, Microsoft, etc.). They land per-provider in future PRs once we have the AGENT_CORE backends for them.
- **Server-side proxy.** Not needed for GitHub. May be needed later for providers that require a confidential secret (then `SECURITY.md` rule "if a confidential secret is structurally required, it lives behind a server-side proxy" applies).
