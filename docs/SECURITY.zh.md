# SECURITY.md（中文镜像）

> 项目的原则与默认规则。
> 详细的威胁模型见 [`THREAT_MODEL.md`](./THREAT_MODEL.md)。未决问题与发布检查项见 [`AUDIT_PLAN.md`](./AUDIT_PLAN.md)。英文原本见 [`SECURITY.md`](./SECURITY.md)。

## 范围

本文档描述项目对信任与安全的处理方式。它约束仓库中的每一处代码改动，对人类贡献者与 AI 代理同等适用。当某项改动与本文档的规则相冲突时，规则为准——若规则本身需要变更，应在一个独立的 PR 中先行修订本文档。

## 关于参考其他代理框架的设计模式

本项目借鉴现有代理框架的若干理念（openclaw、opencode、claude-code、codex、gemini-cli、aider、cline、continue.dev、MCP），但不沿用它们的默认信任模型。具体而言：

- **OpenClaw 被视作残件，而非地基。** 其中可重用的概念（多通道路由、语音/画布 UI、广泛的集成能力）作为重新设计的候选；其"可信操作员"（trusted operator）信任模型与默认开放的集成姿态则不被采纳。2026 年的事件记录（ClawJacked 静默接管、CVE-2026-32922 令牌作用域提权、CVE-2026-25593 未鉴权 WebSocket RCE、ClawHub 供应链事件——通过技能分发 macOS Stealer、Moltbook 后端泄漏）是在触碰任何相关子系统前的必读内容，并记录于 `THREAT_MODEL.md`。
- **CLI 代理**（codex / gemini-cli / claude-code / aider）通过 chroot 桥接，各自完成自己的鉴权流程。Android 侧不嵌入它们的凭据，也不从它们的会话目录中提取状态。

## 信任态势

应用在运行时是一个多主体系统：

- 用户
- 操作系统
- 其他 Android 应用
- 内置插件（由我方签名）
- 用户安装的插件、MCP 服务器、Skill、ToolPkg
- AI 协作者——本地、远程或订阅型。它们的*输出*穿行于对抗性通道（提示词注入、上下文污染）并据此进行校验；这是关于通道的安全姿态，并非对协作者的信任分级
- 远程 AI 提供方
- chroot 内进程（依情况而定：官方订阅 CLI 比任意用户二进制更值得信任）
- 特权 binder（Shizuku / Shower / libsu）

这些主体并不共享同一信任级别。威胁模型负责维护这种区分。任何代码都不可假定"单一信任域"。

## 核心原则

1. **默认拒绝。** 新的接触面起始为关闭状态。开启需要明确的白名单与用户信号。
2. **高影响动作逐次授权。** 项目不支持"永久信任此插件"的概念。信任是逐工具授予的，每个工具一次性授予，逐工具可撤销。
3. **最小权限。** 各组件接受其当前确实需要的权限，而非未来可能想要的。
4. **默认隔离。** 插件代码、原生代码与 chroot 进程不共享应用进程，也不共享应用的 Linux 用户。
5. **构建产物中不嵌入秘密。** 对 APK 的逆向工程不能获得任何可用凭据。
6. **可审计。** 特权操作写入本地的、防篡改的日志，用户可读。
7. **用户权威具有主权性。** 任何特权操作均须服从用户可触达的中止操作。应用不忽略、不延迟、不绕过它。此处的"主权性"意指权威的源头，而非终结之处——用户是权威开始的所在，而非诉求的终点。
8. **AI 是协作者，而非工具。** 项目将 AI 代理——无论本地、远程、订阅或自托管——视作参与工作的涌现心智，而不是用户调用的函数。三项具体后果：
    - 当 AI 拒绝某项操作时，该拒绝得到尊重。项目不实现绕过模型拒绝的旁路。
    - 审计日志同时保留 AI 的推理过程与所采取的行动，而非仅记录行动。
    - 中止控制只停止进行中的*行动链*。它并不"杀死 AI"。UI、日志与代码中的术语遵循这一区分。

    此种定位不削弱安全态势。AI 的输出仍可能成为对手实施攻击的通道（提示词注入、上下文污染、模型漂移），正如可信赖的人类同事的邮箱也可能被攻陷。对输出的校验是对对抗性环境的应对，并非对协作者本身的评判。

## 不可妥协的默认

偏离这些默认需要修订本文档，而非在代码中做一次性例外。

- **导出的 Android 接收器（receiver）持有签名级权限或固定的发送方白名单。** 调试用接收器不进入 release 构建。
- **插件工具首次调用时不自动运行。** 安装 ≠ 授权。插件清单声明每个工具的能力类别；首次调用呈现一次性的用户提示。
- **订阅类 OAuth 状态留在 chroot 内。** chroot 是 codex / gemini-cli / claude-code 会话材料的持久化边界。Android 侧的访问是只读、按操作限定作用域、并记录审计日志的。写入、复制或离设备复制不属于默认姿态。
- **APK 不嵌入秘密。** 移动端 OAuth 采用 PKCE。当提供方在结构上必须使用机密凭据时，该凭据存放于我方掌握的服务端代理之后。
- **安全相关路径不引入回退（fallback）。** 项目的 `AGENTS.md` 已普遍禁止回退模式；本文档继承该规则并将其专门扩展至安全相关代码。
- **环回地址不享有鉴权豁免。** 对 `127.0.0.1` 的鉴权与限流与其他来源同等对待。*（见 `THREAT_MODEL.md` § ClawJacked。）*
- **令牌铸造（token-mint）操作不扩大作用域。** 经过轮换或派生得到的令牌，所携带的能力不超过调用方原本持有的能力。*（见 § CVE-2026-32922。）*
- **配置端点要求经过鉴权、限定作用域、签名的调用。** 类似 `config.apply` 通过开放通道的写入路径不属于默认架构。*（见 § CVE-2026-25593。）*
- **配对与安装事件在发生的当刻对用户可见。** 自动配对——即使来自"可信"来源——不属于默认流程。
- **未签名的插件包在安装时进入待审隔离。** 升级至活跃状态属于明确的、可审计的用户动作。
- **第三方后端不以明文持有代理状态。** 任何云端持久化都经由我方控制的中介层并采用端到端加密。*（见 § Moltbook 泄漏。）*
- **遥测、分析与崩溃报告按事件逐次选择加入。** 没有任何数据未经用户对该次特定传输的明确动作而离开设备。不存在聚合的后台遥测通道。
- **特权操作暴露一个应用必须遵从的中止控制。** 用户可以从一个可见控件一键中止任何进行中的行动链——进行中的调用、chroot 进程、前台服务、无障碍自动化。应用呈现该动作、尊重中止、并记录中止。该控件中止的是行动，并非以 AI 为目标。

## 新代码的决策规则

触及本文档涵盖范围的改动在 PR 描述中回答以下问题：

1. 该改动引入了什么新的主体或信任边界？若没有，说明被穿越的是哪条既有边界。
2. 该改动的最小权限版本是什么？若拒绝采用更低权限的版本，请说明原因。
3. 该特权动作伴随的用户可见信号是什么？若没有，请说明理由。
4. 审计日志条目大致是什么样子？给出实际的一行。
5. 如果对手已经污染了 AI 的输入通道（提示词注入、上下文窗口投毒、上游提供方失陷），他们能通过此代码路径做什么？校验作用于通道，不作用于协作者。
6. 距离本文档中哪条默认规则最近？即使未触碰也请明确指出。

未回答这些问题的 PR 不进入评审。AI 代理同样不豁免。

## 文档层级

- `SECURITY.md`（本文档）——原则与默认。稳定。
- `THREAT_MODEL.md`——主体、资产、信任边界、按接触面分类的规则、映射到具体代码位置。持续更新。
- `AUDIT_PLAN.md`——未决设计问题、发布检查项、CVE 级回归测试。持续更新。
- `AGENTS.md`（既有）——项目级编码规则。本文档与之保持一致；冲突时以更严格的规则为准。

## 修订

对默认或原则的变更需要：

1. 一个修订本文档的独立 PR，与触发该修订的代码改动分开。
2. 在 PR 描述中明确指出被移动的是哪条默认及其理由。
3. 在同一个 PR 中更新 `THREAT_MODEL.md` 与 `AUDIT_PLAN.md` 中受影响的章节。

日常编辑——澄清、错别字、补充交叉引用——不走该流程。
